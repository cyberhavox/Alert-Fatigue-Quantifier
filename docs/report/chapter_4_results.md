# Chapter 4: Implementation and Results

This chapter documents the execution of the Alert Fatigue Quantifier (AFQ) pipeline, presenting empirical outcomes from synthetic logs, machine learning validation metrics, performance profiling statistics, and dashboard components.

---

## 4.1 Synthetic Dataset Statistics
To evaluate the pipeline, we generated a synthetic dataset spanning 30 days for four simulated analysts, yielding **54,081 records** saved to `data/raw/`. The logs mimic realistic operational periods:
* **Nominal Periods:** Alert arrival rates follow a Poisson distribution with $\lambda = 15.0$ alerts per hour. False positive (dismissed) rates represent ~80% of alerts, and average notes length centers around 120 characters.
* **Fatigued Shifts:** High-intensity spikes increase alert rates to $\lambda = 25.0$ alerts per hour. False positive rates surge to ~95% as analysts shortcut verification steps, median enrichment actions drop from 6.0 to 1.2, and empty notes increase.

## 4.2 Machine Learning Model Performance
The Random Forest Classifier was trained on the engineered feature matrix. To ensure generalizability, we performed a Stratified 70/30 Train-Test split and a 5-Fold Stratified Cross-Validation on the full dataset.

### 4.2.1 Performance Metrics
The model achieved the following performance metrics:

| Metric | 70/30 Split Result | 5-Fold Cross-Validation (Mean) |
| :--- | :---: | :---: |
| **Accuracy** | 99.92% | 99.92% |
| **Precision** | 90.24% | 90.56% |
| **Recall** | 99.11% | 98.40% |
| **F1-Score** | 94.47% | 94.27% |

*The high recall (98.40%) is particularly significant for SOC safety, as it minimizes false negatives—meaning the system successfully flags almost all instances where an analyst is about to cross the high fatigue threshold.*

### 4.2.2 Verification of Data Leakage Prevention
During the automated integration test run, we explicitly verified the integrity of the data splits:
1. **Disjoint Sets in Train-Test Split:** The intersection of indices between the 70% training set and the 30% testing set was checked:
   $$\text{Train}_{\text{idx}} \cap \text{Test}_{\text{idx}} = \emptyset$$
   This confirmed that no test samples leaked into the training set.
2. **Disjoint Sets in K-Fold Cross-Validation:** For every one of the 5 folds generated by the cross-validation routing, the validation index set was matched against its training index set, yielding zero overlap.

## 4.4 Streamlit Dashboard Interface walkthrough
The front-end user interface is implemented as a Streamlit web application. It applies custom CSS variables to match modern dark-mode standards (deep charcoal background `#0D1117`, container boxes with glassmorphism borders, and monospace font typography).

The dashboard contains the following UI sections:
1. **Analyst Card Grid:** Renders cards for each analyst displaying their ID, current role, active queue, 60-minute alert closure count, latest computed AFI score, and current fatigue state. Cards are color-coded based on severity:
   - **Nominal:** Slate Grey (`#8B949E`)
   - **Elevated:** Amber/Gold (`#D29922`)
   - **High:** Orange (`#F0883E`)
   - **Critical:** Crimson Red (`#F85149`) with a pulsing hover micro-animation.
2. **Signal Trend Charts:** Renders full-width line charts of rolling signal metrics (triage intervals, enrichment actions) over time using Matplotlib, enabling managers to visually trace the onset of fatigue.
3. **Degradation Anomalies Log:** Displays a table of flagged anomalies where the Mann-Whitney U test p-value dropped below $0.05$, indicating a statistically significant drop in triage quality.
4. **ML Forecasting Timeline:** Displays a timeline of predicted fatigue risks for the upcoming two hours, allowing proactive rebalancing.
5. **Sidebar Recommendations Panel:** Lists targeted advisory suggestions (e.g., *"Recommend a mandatory 20-minute break"*) alongside the mandatory safety disclaimer:
   > **Disclaimer:** All recommendations are advisory. Operational decisions rest entirely with the SOC manager.
